# 翼德 · UI 线稿(mockup / 线稿 / 画个界面 / wireframe)

把勾哥一句话的界面需求,变成一张**所见即最终尺寸、干净如成品**的高保真视觉线稿——
浏览器里 1:1 看真机效果、组件可点着留言,对齐了照着搭 UI。
> 这一步在 `plan` 里(功能带界面时)**自动**走;也可单独喊"翼德 线稿 …"。

**核心原则:给勾哥看的是"界面长什么样",不是规格书。** 一张**纯净的成品级视觉图**——精度直接**长在图里**(每个组件按 kit 实测的最终尺寸/位置/颜色/字号画),勾哥一眼能判设计。**不在图旁列 px/字段/prefab 的 spec 表**(那对看设计的人没意义,实现时去 kit 实测文档查)。糙图不收;画不了的真实美术只用**风格到位的占位**示意位置。

## 怎么做

### 0. 先拿到项目的"真值",再动手
- 读 `~/.yide/projects/<当前项目>.md` 的「🎨 UI/UX 设计预设」(屏幕方向 / **基准分辨率·安全区** / UI 框架 / 美术基调 / 控件约定 / **确认线稿文件夹** / 已确认线稿)+ `unity-context` 技术事实。
- **数值的绝对标准 = 项目 UI kit 的实物**,不是文档。凡需要定具体数值的地方(组件尺寸 · 间距 · 字号 · 配色 · 圆角 · 描边 · item slot 大小 · 按钮高度…),**仔细对照 kit 的 preview 图 + demo scene,以它们为准**——UI 设计规范文档是从 kit 二手提炼的,**冲突一律以 kit 实物为准**。先去 kit 的 `Preview/` 整屏成品图找最接近的同类屏照抄布局比例,再进 demo scene 量真实 RectTransform 的 px/锚点/间距,组件 sprite 看 `ResourcesData/Sprites/Components/`。(具体 kit 与路径见项目档案预设 / ER 专属 prompt。)
- **kit 实物的标准不止尺寸,还含布局/信息架构(IA)**:先在 kit `Preview/` 找最接近的同类屏,**照它整体结构来**——谁是凸起的主入口、底部 nav 放哪几个核心、哪些功能进 nav vs 进侧 rail、banner/hero/CTA 各落哪。**别用设计规范文档的清单或自己想当然定 IA**(踩过:底部 nav 抄文档的 tab 列 → 错;kit 实际是主入口中心凸起,而 Social/Quest/Clan/Friends 全走侧 rail 不进底 tab)。
- **内容来自现游戏,布局/设计来自 kit**:从现有游戏/代码提炼"这屏要展示什么"(内容项、要映射的字段),但**布局与控件外观一律照 kit 复刻,别被现有 prefab/界面的现状带跑**。
- **UI 设计规范文档**(项目内,如 ER 的 `UI_Visual_Design_Guidelines.md`)只作快速索引(调色板 hex · 字级表 · 间距栅格 · 控件标准尺寸 · 安全区 · 动效参数);取值后**回 kit 实物核一遍**。规范与 kit 都缺的项 → 退手游通用最佳实践(触控热区 ≥48dp、最长文案不溢出、一屏一个主 CTA),并提议补一条进项目档案预设。
- 从勾哥描述提炼:这是什么屏(主菜单 / 战斗 HUD / 背包 / 结算 / 单个 widget…)、有哪些主要元素、最重要的操作是什么。
- **预设里缺的关键项(尤其屏幕方向、基准分辨率、确认稿文件夹)→ 用 AskUserQuestion 问勾哥一次,写回项目档案**(学一次以后就有)。拿不准别瞎编(红线②)。

### 1. 写一张自包含 HTML(写到 `<项目>/QA/mockup-<功能>.html`)
**一个文件搞定**,行内 CSS,不引外部资源、不依赖网络。版本号 `window.YD_MOCK_V='<功能>-<日期>-rN'`,按反馈改完必 bump rN。

**只有手机框一个主体,居中放大铺满**(无 spec 边栏、无角标)。

#### 1A. 手机框 —— 1:1 真机、可丈量、纯净成品
- **锁基准分辨率**:device 用项目基准分辨率的真 px(如竖屏 1080×2340),配一小段 `fit()` 按可视区**整数/固定比例缩放**铺满(别缩角落留白),`load`/`resize`/`requestAnimationFrame` 各跑一次(初次布局可能拿到 0)。缩放后框内任何元素都能按比例量出真 px。
- **画安全区 + 落位分区参考线**:顶 status bar、底 gesture bar、tab bar 的安全区线;叠一层屏幕分区(顶栏 / hero / 内容 / 拇指动作区)。**主 CTA 必须落在底部拇指区**,线稿要证明它在那;关闭/返回钮在安全区内。
- **每个组件按最终尺寸画**:按钮、item slot、icon、banner、卡片、3D/角色预览框——全部真实尺寸与间距,**间距吸附到项目栅格**(8 的倍数)。**触控热区**视觉可小但 hit rect ≥ 项目标准(无则 48dp),需要时把热区框示意出来。
- **按钮横排,不竖排**:并排按钮等宽等高等间距。除非是真正的纵向列表(背包行、设置项),否则别竖着堆按钮。
- **主 CTA 尺寸照 kit 实测,别想当然拉全宽**:踩过——把 PLAY 做成近全宽是错的,kit 的 hero CTA 多是**居中、约半屏宽**(如 Button_Tapered)。一切按 kit 实测值,别凭"主按钮就该满宽"的直觉。
- **用真调色板上色,杜绝纯灰占位块**:组件用 UI 设计规范真 hex,**一屏一个主 CTA**(主色),次级/三级/destructive 按规范分明,视觉重量看一眼就分得出主次。稀有度/货币用对应真色。
- **文字用真文案 + 真字级**:禁止 `XXX`/lorem;每段文字套字级表的档;**把"最长态"画进去**(最长名字、0、超大数字按缩写规则如 42.5K/1.2M),证明不溢出不截断。**字体跟项目主字体**;若项目字体不含中文字形(如 CookieRun),所有 label 用英文,别在框内出现会变豆腐的中文。
- **icon / banner / 3D 都要 placement**:不留白、不省略——用风格到位的占位框按真实比例摆好(它最终是什么,看一眼就懂)。
- **框内零文字备注、零角标**:线稿框看起来就是成品,框里只有 UI 本身。每个组件加 `class="yd-region"`(可选 `data-label`,供批注层识别),但**不画可见编号/标签/注释**。
- **数据真实性 = 最长态**:文案/数字都用会出问题的极端值(最长名字、0、超大数缩写)画进去,证明不溢出——精度就靠这个体现,不靠旁边写字。

> **可选**(勾哥要才加):需要确认交互态时,在手机框下方补一条**状态条**,把关键件各态画一遍(normal/pressed/disabled/空态/极端值)。默认**不加**,保持纯净。

### 2. 嵌入批注层(给勾哥点选留言用)
把 `${CLAUDE_SKILL_DIR}/templates/mockup/annotate.html` 的内容**原样**贴到 `</body>` 之前——勾哥点某组件就弹框写意见、边打边自动保存、一键复制反馈。这是**反馈通道**,与"框内零备注"不冲突。

### 3. AI 自验证 —— 自己进浏览器看,改到满意再给勾哥(铁律,别甩给他当第一个测试者)
**别"写完就弹给勾哥"**。HTML 写完先自己用 headless 浏览器渲染成 PNG、用 Read 看真实渲染、当 art director 挑毛病(对照 kit 实物 + §1A 清单)、改、再截,**循环到自己满意**才进下一步。
1. **找本机 headless 浏览器**(任一存在即可,优先 Chrome):
   - Win:`C:\Program Files\Google\Chrome\Application\chrome.exe` 或 Edge `C:\Program Files (x86)\Microsoft\Edge\Application\msedge.exe`
   - mac:`/Applications/Google Chrome.app/Contents/MacOS/Google Chrome`;Linux:`google-chrome`/`chromium`
   - 都没有 → 老实告诉勾哥"截不了图,这张没自验证",直接进第 4 步让他看。
2. **截图**(Bash;路径有空格用引号):
   ```
   "<chrome>" --headless=new --disable-gpu --hide-scrollbars --force-device-scale-factor=2 \
     --window-size=560,1480 --screenshot="<项目>/QA/_mock_shot.png" "file:///<html 绝对路径>"
   ```
   - headless 截的是 **window-size 区域**(非整页)。窄而高的 window 让 `fit()` 把手机框放大看清细节;但 **window 宽度必须 > 缩放后手机框宽**(踩过:window 640 把 ~670px 宽的手机框右侧切掉)——取 **800×1480** 左右最稳。
   - `force-device-scale-factor=2` 出 2× 高清图,字和描边看得清。
3. **Read 那张 PNG**,两层挑毛病:
   - **执行层(像不像 kit)**:布局/尺寸/落位对不对、有没有**溢出/重叠/死空间**、casing/配色/字级跟不跟 kit、手机框宽有没有被切。
   - **producer 层(摆得对不对)**:信息层级清不清、**留存钩子**(每日/邮件/活动该常驻露出带红点,别埋进汉堡)、**变现入口**(Shop/Offer/BP 首屏露不露)、**拇指区经济**(最值钱的位给最高频/最赚钱的,别给低价值功能)、**底部 nav 留核心循环 + 凸起主入口**、engagement 类(任务/成就/社交)收进 tab 让红点冒泡。拿不准可对照同类竞品(Archero/Habby 等)的 home 范式。
4. **有问题就改 HTML(bump rN)→ 重截 → 再 Read**,循环。满意了删掉临时 `_mock_shot*.png`。
> 这是线稿版的 ER 收尾铁律:自己测到位再交付。改 HTML 是纯文本、零风险,放手迭代别问。摆位的 IA/产品取舍若是勾哥该拍的方向(放哪几个、走 nav 还是 rail)→ 用 AskUserQuestion 摊给他,别替他定。

### 4. 自动弹给勾哥看
自验证满意后:`node -e "require('${CLAUDE_SKILL_DIR}/scripts/lib.js').openUrl('<那个html的绝对路径>')"`(`YIDE_NO_OPEN=1` 时跳过,改为告诉他绝对路径让他自己开)。**老实交代**:哪些是自验证过的、哪些没法验(如真实美术观感)。

### 5. 确认闭环
勾哥要么直接说"3 号挪到左上",要么在 mock 上点区域 → 弹框写意见 → 点「复制反馈给翼德」粘回来 → 你据此改 mock(bump rN)→ **先自截自看(回第 3 步)** → 再弹 → 直到他说"对了"。**确认后才进实现**。

### 6. 存确认稿(供 yide / 其他 AI 以后提取复用)
勾哥说"对了"后——
- 把这张(连批注层)存到项目档案记录的**确认线稿文件夹**;默认 `<项目>/Docs/UX/mockups/<功能>-<日期>.html`。**该文件夹首次使用时用 AskUserQuestion 问勾哥确认路径**,写进项目档案「确认线稿文件夹」字段。
- 文件夹**首次创建**时放 `README.md`:"这是勾哥确认过的高保真 UI 线稿(所见即最终尺寸的纯净视觉图),做新界面前可参考既有布局/控件约定/真实尺寸。"
- 在项目档案「已确认线稿」追加一行:`<功能> → <路径>`。
- 工作中未确认的草稿留 `QA/`,**别和确认稿混**。

## 原则
- **自己看过才交付**:写完先 headless 截图自看、迭代到满意(§3),别让勾哥当第一个测试者;截不了图就老实说"没自验证"。
- **给勾哥看设计,不给规格书**:一张纯净成品级视觉图,精度长在图里(按 kit 实测 px 画),**不附 spec 边栏/角标/字段映射**——那些实现时去 kit 实测文档查。
- **框内即成品**:手机框是唯一主体,居中放大;框里只有 UI 本身,对得上栅格、量得出 px。
- **kit 实物是标准,IA 也算**:布局/放哪几个/谁凸起/进 nav 还是侧 rail 都照 kit 实物核,别用设计规范文档清单或想当然;内容来自现游戏,布局来自 kit。
- **摆完用 producer 眼睛再扫一遍**:不只"像不像 kit",还要信息层级/留存钩子/变现入口/拇指区经济站不站得住;拿不准的产品取舍摊给勾哥。
- **状态按需**:默认纯净;勾哥要确认交互态时才在框下补状态条。
- **诚实**:真实美术资源、动效细节画不了就说画不了,只用风格到位的占位示意位置。
