# 翼德 · 对抗式评审(review / 评审 / 挑刺 / 审一下)

收尾时让一个**全新上下文**的评审员只盯 diff 挑硬伤——专治"自己写的代码自己看不出问题""只为过测而写测"的自评盲区(charter #4 的牙齿)。
分工:**review 只挑正确性与耦合,不挑风格**;风格/Unity lint 归把关器(PostToolUse),测试计划/bug SOP 归 `qa`。三者别互相抢活。

## 怎么做
1. **取 diff**(只把改动给评审员,**不给写代码时的对话**——避免被同一思路带偏):
   - 默认:`git diff` + `git diff --staged`;给了 base 就 `git diff <base>...HEAD`。
   - 必要时附被改文件的关键上下文;diff 太大就**按文件分批**评审,别截断了还假装看全了。
2. **派新上下文 subagent**(用 Agent/Task 工具起一个 fresh context 的评审员),交给它这套**对抗式**指令:
   > 你是独立代码评审员,立场是**主动找出这段 diff 里真实会出问题的地方**。你只看下面的 diff,没有写它时的对话背景。
   > **只报这三类,且每条都要站得住**:
   > ① **正确性 bug** —— 逻辑错 / 边界与空值 / 竞态 / 错误处理缺失 / 破坏了原有行为;
   > ② **耦合与暗坑** —— 新依赖方向错、改这里悄悄波及别处、隐藏状态、跨模块假设失效、生命周期/时序坑;
   > ③ **验收标准或红线被违背**。
   > **不要报**:命名、格式、风格、"可以更优雅"——不归你管。
   > 每条按 `严重度(高/中/低) · 文件:行 · 问题 · 为什么是真问题 · 怎么验证或修` 给。没十足把握标"待确认",**别凑数**;真挑不出就直说"未发现正确性/耦合问题",别硬编。
3. **有 Unity MCP** → 评审员可调 `read_console` / `run_tests`,把"我觉得会崩"变成"我看到它崩",以引擎真值为准。
4. **汇总给勾哥**:按严重度列出;**高**的建议改完再交付。翼德只摆问题,不替勾哥拍板改不改。

## 原则
- 新上下文 + 只看 diff + 对抗立场 = 破自评循环(同一上下文里自评容易自我确认)。
- 只挑正确性/耦合,把风格让给把关器,别两头都管显得啰嗦。
- 诚实第一:没真跑别说跑过了;挑不出就老实说没问题,不为"显得有用"硬找茬。
