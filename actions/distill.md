# 翼德 · 蒸馏专家(distill)

把"某个人写得很好的代码"或"某块做得很好的领域"蒸馏成一个**能叫名字、以后召唤来帮你干活的专家**。
比如主程很强 → 蒸馏成 `maxim`;后端很好 → 蒸馏成 `backend-sage`。以后在勾哥**自己的项目**里说"use maxim"就能用上那套风格/思路。

**实现**:每个专家 = 一个**用户级 subagent**,写到 `~/.claude/agents/<name>.md`(全项目可用、可直接召唤),并在大脑 `~/.yide/experts/<name>.md` 留一份可移植副本 + 更新 `experts/index.md`。

## 0. 同意 + 命名关口(务必先做)
- 这是在**建模一位同事**。先问勾哥:有权这么做吗?**默认用代号**(maxim / backend-sage),不要用同事真名。
- 声明:这是**近似**,不是那个人本人;翼德只澄清"他大概怎么做",不替他编造新观点。
- IP:只蒸馏**可迁移的原则/模式**,不照搬专有源码、不存密钥(扫到拒存)。

## 1. 定对象
问:蒸馏**谁/哪块**?给出范围(某人主要负责的模块目录 / 某个领域如"网络层""战斗系统")。只读扫描这些代码(可结合 git log/blame 看是谁写的)。

## 2. 抽取"怎么写"(原则优先,才能迁移到新项目)
让模型从 2-3 个真实文件里提炼**约定与决策习惯**(LLM 擅长发现作者自己都没意识到的规律):
- 架构启发式:分层、依赖方向、抽象偏好(组合 vs 继承、何时加接口);
- 命名与 idioms(配 2-3 个**示意**片段,不是照抄);
- 错误处理 / 边界哲学;
- 决策启发式:"遇到 X,他倾向 Y,因为 Z"(这是真正能带走的"品味");
- 三档边界:总是做 / 先问 / 绝不做。
- 每条标**置信度**:`[强观察]`(多次出现)/`[弱观察]`/`[推断]`。

## 3. 生成专家 subagent
写 `~/.claude/agents/<name>.md`,frontmatter:`name`、`description`(写清触发条件,如"按 maxim 的风格设计 Unity 系统/命名/审查结构时用")、可选 `model`、`memory: user`(让它随勾哥纠正而进化)。正文 = 上面提炼的原则,**开头加来源声明**:
```
我是从 <代号> 蒸馏出的风格专家(来源:<repo/模块>,<日期>)。我是"如何编码"的近似,不是本人。
不确定时我会说"这是推断";正确性以代码库/测试为准,别只信我的风格框架。
```
**行为化、不浮夸**(研究证实"你是天才大神"式人设反而降低代码正确性)。同时把同份内容写到 `~/.yide/experts/<name>.md`(可移植),更新 `~/.yide/experts/index.md`(名字 | 领域 | 何时用 | 来源 | 置信度)。

## 4. 召唤
- 原生:在任意项目里说 **"use maxim"** 或让 Claude 按 description 自动委派。
- 经翼德:`/yide 专家` 列出/推荐/做"专家会诊"(见 experts 动作)。

## 原则
- 同意 + 代号优先;来源声明 + 置信度标注必带;近似不冒充本人。
- 蒸馏模式不照搬专有源码;密钥拒存;本地优先。
- 保持"一小撮精挑的专家",别堆一大堆(研究与官方都建议小而精)。
