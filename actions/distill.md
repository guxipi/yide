# 翼德 · 蒸馏同事(distill)

把公司里多人写的代码/架构,蒸馏成勾哥**私人、可移植、可复用**的知识库,存到 `~/.yide/distilled/`:
`architecture/`(架构图与"怎么运作")、`patterns/`(去耦后的可复用模式)、`decisions/`(设计取舍)、`_raw/`(原始片段·高风险区)、`index.md`。

## 0. 先过 IP 关口(每个新源仓库做一次,务必)
- 提示用户(**非法律意见**):带走"通用知识/可迁移模式/你自己的心得"通常可以;**逐字搬运专有源码、密钥、雇主特有的业务逻辑**有风险,而且"带走文件"是法律上最重的违法因子。建议查一下你的雇佣/IP 协议。
- **默认只蒸馏知识与去耦模式**(再表达,不照抄)。要不要连原始代码片段一起留(尺度 2)?→ 见第 5 步,逐条确认。
- ⚠️ 若你的 `~/.yide` 放在 Google Drive 等云同步盘:蒸馏内容(尤其 `_raw/` 原始片段)会被同步上云。涉及雇主代码时,建议改存**本地不同步**的位置或单独账号。先问清楚再写。

## 1. 定范围
问用户:蒸馏哪个 repo / 哪些目录?目标是"看懂架构"还是"带走某几个可复用模块"?

## 2. 架构地图(architecture/)
- 只读扫描:模块/依赖关系、入口点、关键流程。
- 产出 C4 风格(Context/Container/Component)+ Mermaid 图 + 每模块"怎么运作"短文,**带 `file:line` 引用回源码**(引用,不是把源码抄进来)。
- 标注置信度;**"为什么这么设计"代码看不出来 → 问用户补**(这是他真正能带走的)。

## 3. 模式目录(patterns/)
- 对每个可复用模式:写成**去耦的通用模板**——改掉公司名/业务常量、删专有算法、重命名为通用类型。
- 每条记:解决什么问题 → 通用解法(模板)→ Unity 手游注意点 → 用户自己的点评。
- **再表达,不是搬运。** 大段专有源码不进 patterns。

## 4. 密钥/秘密硬拦
- 扫到凭证、API key、token、内网地址、专有常量/算法 → **拒绝写入**,提示用户这类不能带。

## 5. ipScope 标注 +(尺度 2)原始片段
- 每条标 `ipScope: 通用 | 专有`;判断准则:行业通用/竞品也容易知道 = 通用;雇主特有 = 专有(给"专有"挂警告)。
- **尺度 2(保留原始片段)**:只在用户**逐条明确确认**时,把原始 .cs 片段存进 `_raw/`,frontmatter 标 `ipScope: 专有 / 原始代码`,并再警告一次(尤其云同步时)。默认不存。

## 6. 写入 + 可查
- 按下面 frontmatter 写文件,更新 `~/.yide/distilled/index.md`(按 来源项目 / 主题 / ipScope 分组)。
- 之后 `/yide 蒸馏 <关键词>` 检索;新项目里可直接调取这些通用模式。

```yaml
---
title: ""
taxonomy: architecture | pattern | decision | raw
ipScope: 通用 | 专有
sourceProject: ""
unityVersion: ""
tags: []
created: ""
---
```

## 原则
- 默认再表达、不照抄;密钥一律拒;`_raw/` 逐条确认才存并强警告。
- 本地优先、不上传第三方;别把雇主代码和个人项目混在同一会话/同一store里乱炖。
- 不确定是否越界 → 停下问用户,别替他冒险(对应红线:不编造、不擅自大改)。
