# 翼德 · 闭环造鸭(plan / 计划 / 闭环造鸭)

把勾哥**几句模糊的话**(一个 feature 或一段 core loop),走完整闭环做成**能在引擎里跑起来的占位可玩切片**:**对齐 → 造 → 验 → 交付**(规范驱动开发)。
卖点要诚实:**只卖"早能上手的竖切 + 逻辑层自动验到绿 + 集成保证不崩 + 手感你来定 + 数值你能自己调"**,不卖"通宵全自动出成品"。
(单文件、可逆、琐碎的活直接做,别走闭环——反过度工程。)

---

## 阶段一 · 对齐(唯一重人工,务必对死;**规范烂→交付烂**)
1. **读 context**:`unity-context` + 项目档案「🎨 UI/UX 设计预设」+ 命中的 lessons 与红线 + 相关源码。
2. **接口盘点**(治"集成进屎山"):列出这功能要碰的现有系统/脚本,**读真实代码确认它们当前的契约/输入输出(别猜)**,标出耦合点;判断能否**隔离开发**。
3. **调研同类做法**(charter #3,带 URL 引用,缓存 `~/.yide/.cache/bp-<主题>.md`;见下方纪律)。
4. **角色镜头快速内联讨论**(别派慢 subagent):UI/UX + 美术总监 + PM 出方案 → 架构总监 challenge → 收敛;只给勾哥**一段"资深小结"**。
5. **带界面 → 出线稿确认**:执行 mockup 动作(读 `actions/mockup.md`),勾哥确认布局再继续。
6. **起草 1 页规范** `<项目>/QA/plan-<功能>.md`:
   - **目标**:满足验收的最小方案。
   - **智能默认**(治"几句话太模糊但别问 20 题"):模糊处**填合理默认值并标 `[假设]`**(依据项目约定/同类做法),让勾哥 yes/改。
   - **验收标准(EARS,且分三层标注)**:每条标 **`[逻辑·自动测]`**(掉率/伤害/计时/状态机)/ **`[集成·冒烟]`**(能生成/不报异常/能走到终态)/ **`[手感·试玩]`**(速度/强度/节奏/平衡)——决定阶段三谁来验。
   - **关键设计**(架构总监:边界/耦合/暗坑,优先低耦合)+ **任务清单**(依赖排序)+ **测试计划**。
   - **只问最高杠杆的 3–5 个 `[需澄清]`**,别淹没勾哥;其余靠 `[假设]`。**红线门**:过 6 条红线。
7. **停下等批准**:勾哥过完 `[假设]` + `[需澄清]` 并点头后,才进阶段二。绝不自动开干。

## 阶段二 · 造(先竖切、隔离、占位)
- **0. 开造前先建检查点(问勾哥,后悔药)**:用 AskUserQuestion 问一句——"要不要先建个安全检查点(`git stash` 存住你没提交的活 + 在专用分支 `yide/闭环-<功能>` 上造),万一失败能一键回滚?"
  - 要 → `git stash -u`(保住未提交的活)→ 记原分支 + 当前提交到 `<项目>/QA/闭环-<功能>.checkpoint` → `git checkout -b yide/闭环-<功能>-<日期>`,之后**每个里程碑一个原子提交**(连同新建的 .meta/场景/prefab 一起);**主分支自始至终不碰**。
  - 不要 / 非 git 项目 → 跳过(非 git 可把"将碰的文件 + 新建清单"备份到 `QA/.闭环-backup/` 兜底),并**明确告诉他"没检查点,回滚得手动"**。
- **先竖切**(治失控/早反馈):先做一个**最小能玩的垂直切片**(如 Boss 只会走 → 能被打死),**先让勾哥 gut-check**,再逐步加技能/空投/接主流程。**别一口气闷到底**。
- **隔离开发**(治集成屎山):新功能先在**独立 / additive 场景**里做、能单独验,**最后一步才接线**进主流程。
- **占位(不做美术)**:① **先复用项目现有 prefab/资源,但只"借皮"**——剥离或 stub 掉它的非视觉脚本、记下它拖来的依赖;② 实在没有才用 primitive 灰盒。一律标"占位待美术"。
- **可调参数集中**(治试玩回流):Boss 血量/速度/技能 CD/掉率 等**抽到一个 config**(ScriptableObject 或 JSON),让勾哥**自己改数值、不用重编译**。
- 守 charter:先定位根因、不 hack、只动该动的、贴合既有风格。

## 阶段三 · 验(oracle 循环 + 护栏)
> oracle = 测试 + **已接的 Coplay(Unity MCP)**,不是新东西。
- **只对 `[逻辑]` + `[集成]` 两层自动验到全绿**;**`[手感]` 留给勾哥试玩,绝不声称机器验了手感**。
- **按层分工(玩法以 PlayMode 为真值)**:纯逻辑(掉率/伤害/计时/状态机)用 **EditMode**(快);**真玩法 / 集成(Boss 关核心)用 PlayMode——这才是真值,以它为准**,别为省时间用 EditMode 假装测了玩法。PlayMode 慢就**批量跑**(一次进 Play 多验几条),靠下面的护栏控时间。
- 跑 `run_tests` / `read_console` / 必要时 PlayMode 模拟操作 → 没过就**自己定位、纠错、重跑**。
- **护栏**(治又慢又脆/失控):设**迭代次数与时间上限**;**卡住就停下,报"卡在哪、试过什么"**,别空转烧时间/token;**Coplay 掉线/超时 → 降级为"交付 + 让勾哥自己跑测"**,不假装跑过。
- 收尾跑 **`翼德 评审`**:新上下文 subagent 只看 diff、只挑正确性/耦合/暗坑。

## 阶段四 · 交付 + 试玩
- **先全绿,才惊动勾哥**:阶段三 `[逻辑]`+`[集成]` 全绿之前**不要**喊勾哥手测/试玩——别把半成品丢给他。
- **真机实测(可选)**:全绿后**问一句**"要不要在安卓真机上实测?(可选)"。要 → guide USB 调试 + adb 取证 + 按 SOP 出**真机 QA 报告**(见 `qa` 动作 F),聚焦真机表现/手感、不混基础 bug;不要 → 直接交付。
- 交一个**早能上手的占位可玩切片**,外加**三样**:
  1. **试玩清单**(治"不信绿"):把验收标准变成一张"照着点一遍"的 try-list,他自己点着确认;
  2. **测试明细**:哪条绿/红、哪层是自动验的、哪层要他试玩;
  3. **占位清单**(哪些等美术)+ **可调参数在哪**(让他自己调手感)。
- **试玩反馈回流**:勾哥说"Boss 太快 / 技能无聊" → 改对应**验收或 config 数值** → **只重跑那一条验证**,不全量重来。

## 失败回滚(闭环造鸭的一部分,不是单独动作)
勾哥说"回滚 / 撤销这次闭环"时:
- 读 `<项目>/QA/闭环-<功能>.checkpoint` → **先给他看"将把整段闭环丢弃、回到检查点"** + 一次确认(破坏性,但就一条命令)。
- **整段当一个单位一次撤**(别逐条 revert,会撤出一堆不稳定中间态):切回原分支 → 弃掉 `yide/闭环-*` 分支 → 恢复 `git stash`(还原他原来没提交的活)。因为全程在独立分支,**主分支没被碰过,回滚干净、别的工作零影响**。
- 回滚前**再 stash 一份当前**(万一回滚也想反悔);回滚后提醒 Unity 重新导入(Library 缓存自愈)。
- 想保留一部分 → `git reset` 到某个里程碑 commit(而非全弃)。
- **没建过检查点** → 老实说只能手动撤 / 靠编辑器历史,别假装能一键回。

---

## 调研的纪律(把阶段一第 3 步做好,而不烦人)
- **默认做、不用问**:非琐碎就查同类做法/最佳实践;琐碎/可逆小改跳过。
- **变化快的主题更要查最新**(URP / Addressables / IL2CPP / 手游 GC·draw call·内存):带 URL 引用 + 标 Unity 版本与日期。
- **缓存** `~/.yide/.cache/bp-<主题>.md`,同主题未过期(默认 30 天)直接复用;**绝不开会话时自动搜**。

## 原则
- **对齐最重要**(规范烂→交付烂);**好玩机器验不了**(靠勾哥试玩);先竖切早反馈、先批准后实现、盲点先问不猜(但用 `[假设]` 少问);占位先借皮 prefab 再灰盒;数值集中让勾哥自己调;**没真验证别说验证过**。
