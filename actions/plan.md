# 翼德 · 闭环计划起草(plan)

当任务是**多文件/架构级/"先想清楚再写"**时,别直接闷头写——先按项目 context 起草一份**最小但闭环**的计划(架构→实现→QA),让勾哥过目批准再动手。
(单文件、可逆、琐碎的活直接做,别为这点事起草计划——反过度工程。)

## 步骤
1. **读 context**(翼德已具备):Unity 版本/管线/Input/Addressables(unity-context)、命中的 lessons 与红线、相关源码文件。
1.5. **先调研同类做法(charter #3:非琐碎功能默认做,不用勾哥提醒)**:同类游戏/项目这功能一般**怎么解**、Unity 手游的**最佳实践/常见坑**是什么。结论**带 URL 引用**给勾哥参考,缓存到 `~/.yide/.cache/bp-<主题>.md`(标 Unity 版本+日期,默认 30 天内同主题复用,不重复查)。琐碎/可逆小改跳过这步。
1.7. **若功能含界面/布局 → 资深讨论 + 出线稿确认(补勾哥单干缺的设计视角)**:
   - **快速内联**走一轮(别派慢 subagent、别让勾哥干等):**UI/UX** 谈布局与流程、**美术总监** 谈风格/性能可行性、**PM** 谈最小可玩与优先级 → **架构总监** challenge 耦合/暗坑 → 收敛出方向。
   - 只给勾哥**一段简短"资深小结"**(每角色一两句 + 架构的质疑 + 最终方向),**别展示全过程拉锯**。
   - 然后执行 **mockup 动作**(读 `actions/mockup.md`)出一张**可批注 HTML 线稿**;勾哥确认布局/流程后,再进第 2 步起草实现计划。
   - 想要更深的独立"会诊"再用 `翼德 会诊`(那个慢些,按需)。
2. **起草** `<项目>/QA/plan-<功能>.md`(项目产物进项目 repo,不进 ~/.yide):
   - **目标**:1–2 行,**满足验收的最小方案**(不过度设计)。
   - **验收标准(EARS,可测)**:`当<事件>,系统应当<行为>` / `若<条件>` / `不应当…`——每条都能写成测试。
   - **关键设计(charter #4:架构总监视角)**:只写真正要定的取舍,给备选、选最简的那个;**显式标出边界 / 耦合 / 暗坑**——谁依赖谁、依赖方向对不对、改这块会波及哪些、有没有更松耦合的做法;优先低耦合方案,拿不准的耦合点列进 `[需澄清]`。
   - **任务清单**:按依赖排序,标出高风险项。
   - **测试计划**:每条验收标准对应哪个 EditMode/PlayMode 测试或手测步骤(复用 `翼德 qa` 的风险排序 + bug SOP)。
   - **`[需澄清: …]`**:每个盲点都标出来,**别猜**。
   - **红线门**:计划必须过翼德 5 条红线(不屎山/不过度工程/不编造API/不加没要的/不擅自大改)。
3. **问澄清**:把 `[需澄清]` 用 AskUserQuestion 问勾哥,更新计划。
4. **停下等批准**:勾哥点头**之后**才写代码。绝不照自己写的计划自动开干。

## 调研的纪律(怎么把第 1.5 步做好,而不烦人)
- **默认做、不用问**:非琐碎功能就去查同类做法/最佳实践(charter #3)。琐碎/可逆小改跳过。
- **变化快的主题更要查最新**:**URP / Addressables / IL2CPP / 手游 GC·draw call·内存** 这类,务必 WebSearch/WebFetch 拿当前实践,**结论带 URL 引用** + 标 Unity 版本与日期。
- **缓存**:存 `~/.yide/.cache/bp-<主题>.md`;同主题且版本/日期未过期(默认 30 天)直接复用,不重复查(省成本)。
- **绝不在开会话时自动搜**(防过时、不烧 token);调研发生在"动手做某功能时",不是开场。

## 闭环 QA(诚实)
- **写好测试计划**;若项目用 CLI 跑测试,加"测试不绿不算完"的约束。
- **收尾跑一次 `翼德 评审`**(charter #4 的牙齿):派**新上下文**的对抗式评审 subagent,只看 diff、只挑正确性/耦合/暗坑,防"只为过测而写测"的自评循环。
- **有 Unity MCP**:翼德能真跑 EditMode/PlayMode、读 Console、自我修正闭环——**以引擎跑出来的真值为准**,别凭"应该没问题"忽悠跑过了;没真验证就如实说没验证。

## 原则
- 最小够用、先批准后实现、盲点先问不猜;查实践要带引用、缓存、不自动。
