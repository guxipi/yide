# 翼德 · 闭环计划起草(plan)

当任务是**多文件/架构级/"先想清楚再写"**时,别直接闷头写——先按项目 context 起草一份**最小但闭环**的计划(架构→实现→QA),让勾哥过目批准再动手。
(单文件、可逆、琐碎的活直接做,别为这点事起草计划——反过度工程。)

## 步骤
1. **读 context**(翼德已具备):Unity 版本/管线/Input/Addressables(unity-context)、命中的 lessons 与红线、相关源码文件。
2. **起草** `<项目>/QA/plan-<功能>.md`(项目产物进项目 repo,不进 ~/.yide):
   - **目标**:1–2 行,**满足验收的最小方案**(不过度设计)。
   - **验收标准(EARS,可测)**:`当<事件>,系统应当<行为>` / `若<条件>` / `不应当…`——每条都能写成测试。
   - **关键设计**:只写真正要定的取舍,给备选、选最简的那个。
   - **任务清单**:按依赖排序,标出高风险项。
   - **测试计划**:每条验收标准对应哪个 EditMode/PlayMode 测试或手测步骤(复用 `翼德 qa` 的风险排序 + bug SOP)。
   - **`[需澄清: …]`**:每个盲点都标出来,**别猜**。
   - **红线门**:计划必须过翼德 5 条红线(不屎山/不过度工程/不编造API/不加没要的/不擅自大改)。
3. **问澄清**:把 `[需澄清]` 用 AskUserQuestion 问勾哥,更新计划。
4. **停下等批准**:勾哥点头**之后**才写代码。绝不照自己写的计划自动开干。

## 可选:查最新最佳实践(仅变化快的主题)
- 只在碰到 **URP / Addressables / IL2CPP / 手游 GC·draw call·内存** 这类变化快的主题时,**问一句**"这块要不要我查下最新实践再写?"。
- 同意 → WebSearch/WebFetch,**结论带 URL 引用**;把结果缓存到 `~/.yide/.cache/bp-<主题>.md`,标注 Unity 版本与日期;下次同主题且版本/日期未过期(默认 30 天)直接用缓存,不重复查。
- 绝不在开会话时自动搜(省成本、不烦人、防过时)。

## 闭环 QA(分阶段,诚实)
- **现在(无 MCP)**:写好测试计划;若项目用 CLI 跑测试,可加"测试不绿不算完"的约束;并可派审查 subagent(`翼德 专家`)独立挑刺、防只为过测而写测。
- **接了 Unity MCP 后**:翼德能真跑 EditMode/PlayMode、读 Console、自我修正闭环——这半截**接 MCP 前是纸上**,如实告诉勾哥,别忽悠跑过了。

## 原则
- 最小够用、先批准后实现、盲点先问不猜;查实践要带引用、缓存、不自动。
